# AI Object Lifecycle — A Domain-Neutral RubyLLM Framework

> The framework view: the reusable engine underneath an app where **a user generates and refines
> objects with the help of an LLM through a polymorphic chat** — stripped of any specific domain.
>
> This is the most abstract of the three LLM design docs:
> - `LLM_ARCHITECTURE.md` — migrating an *existing* app toward separated delivery paths.
> - `LLM_BLUEPRINT.md` — a *greenfield* reference, with a worked example.
> - **`AI_OBJECT_LIFECYCLE.md` (this doc)** — the *engine* extracted: what's generic vs. what an app declares.
>
> Grounded in `ruby_llm` features verified at <https://rubyllm.com/> (June 2026). Code is
> **illustrative**, not drop-in.

---

## 1. The domain-neutral loop

Every "generate & refine with an LLM" app is the same four-beat loop, regardless of domain
(content, contracts, recipes, workouts, code, …):

1. **Open** a chat with a `purpose` = `"<verb>_<type>"`, anchored to a context — a *parent* when
   generating, the *object itself* when refining.
2. **Converse** — free-form, context-aware: the system prompt is assembled from context providers;
   optional tools let the model pull more.
3. **Commit** — a structured `schema` extraction turns the conversation into attributes.
4. **Persist + redirect** — create (generate) or update (refine) the target; optionally keep refining.

In one line: **conversational create/update of any resource, with read-context assembled
automatically.** Everything domain-specific is *data the app registers* — never engine edits.

---

## 2. The engine ↔ app boundary

| The engine provides (generic, written once) | The app declares (per noun) |
|---|---|
| Polymorphic `Chat` + `purpose` + anchor | Which models are AI objects (`include AiObject`) |
| Purpose → agent → plan **dispatch by convention** | A `schema` per object |
| Generate / Refine **verbs** (shared extraction + persist) | Optional custom agent prompts (else defaults) |
| Pluggable **context providers** + ordering | Which providers apply (`context:`) + `parent:` |
| Streaming, tool-call persistence, cost/token tracking | Domain tools (optional) |

The whole point: **adding an AI capability = declaring a noun and a schema.** The engine is closed
for modification, open for extension.

---

## 3. The grid (the organizing principle, domain-free)

Every prompt contribution is generated by two axes — **facts vs. directives** × **developer-owned
(code) vs. creator-owned (data)** — and the engine guarantees each cell has **one owner and one
delivery channel**:

| | **Developer-owned** (code) | **Creator/User-owned** (data) |
|---|---|---|
| **Facts** | *(rare; folded into role preamble)* | context-provider segments (object + ancestry + profile) |
| **Directives** | agent prompt templates (per purpose) | user/object directive columns (own segment) |

Delivery channels (separate paths, never concatenated except by the composer):
agent `instructions` (system message) · `schema` → `response_format` · pushed facts · **tools** (pull)
· labeled directive segments.

## 4. Generalization 1 — nouns self-register (convention over a hand-written registry)

A model opts in; the engine derives purposes, agents, and the plan. No registry table to maintain.

```ruby
# ILLUSTRATIVE
class Article < ApplicationRecord        # could be Recipe, Contract, Workout, …
  include AiObject
  ai_object schema:  ArticleSchema,
            parent:  :workspace,         # optional context anchor for generation
            context: [:user_profile, :parent]   # which providers feed the prompt
end
```

From that one declaration the engine knows:

- **Purposes** `generate_article` / `refine_article` exist (registered on the `Chat` enum).
- **Agents** resolve by **naming convention** — `Articles::GenerateAgent` / `Articles::RefineAgent`
  if defined, else a generic `Ai::GenerateAgent` / `Ai::RefineAgent` parameterized by the schema.
  (Convention with an override hook.)
- **Plan** is derived: generate → `parent.articles.build`; refine → `object.assign_attributes`;
  redirect → `polymorphic_path(record)`.

Conceptual concern shape:

```ruby
# app/models/concerns/ai_object.rb  — ILLUSTRATIVE
module AiObject
  extend ActiveSupport::Concern

  class_methods do
    def ai_object(schema:, context:, parent: nil)
      AiObject::Registry.register(self, schema:, context:, parent:)   # adds purposes + plan
      has_many :chats, as: :chattable, dependent: :destroy
      @ai_schema, @ai_context, @ai_parent = schema, context, parent
    end
    attr_reader :ai_schema, :ai_context, :ai_parent
  end
end
```

## 5. Generalization 2 — context as pluggable providers, not a fixed ancestry

The most domain-specific assumption to remove is a *linear* parent chain. Generalize "facts" to a set
of **`ContextProvider`s** — each a tiny object with `relevant?(anchor)` and `segments(anchor)`:

| Provider | Yields | Note |
|---|---|---|
| `SelfContext` | the object's own current attributes | **only at refine** (generate has no object yet) — this is why the verbs share code |
| `ParentContext` | declared `parent:` facts (one, many, or none) | flat objects are fine |
| `UserContext` / `ProfileContext` | user-scoped persona / preferences / memory | the "brand"/"voice" cell, app-agnostic |
| `RetrievalProvider` | large / optional context (RAG, recent objects, web) | exposed as a **tool** (pull), not pushed |

```ruby
# ILLUSTRATIVE
class ContextProvider
  def relevant?(anchor) = true
  def segments(anchor)  = []          # => [Segment, …], labeled blocks
end
```

A composer runs the declared providers **in order**, collects labeled `Segment`s, and renders the
system prompt. This replaces the hardcoded walk: an app can have a tree, a DAG, multiple profiles, or
no hierarchy — it just lists providers. Ordering and precedence stay a *declaration*, not call-order,
and are unit-testable without an LLM.

## 6. Generalization 3 — verbs as one shared engine

Two thin, domain-neutral entry points that share extraction and context assembly:

```ruby
# ILLUSTRATIVE
Ai::Generate.call(type: Article, parent: workspace, transcript:, user:)   # build → extract → save
Ai::Refine.call(object: article, transcript:, user:)                      # extract → update
```

Dispatch is `chat.purpose → (verb, type)`. The only real differences between the verbs:

- **which agent's directives apply** (`generate` vs. `refine` prompt), and
- **whether `SelfContext` emits anything** (it does only when an object exists).

Both are handled by the provider list and the agent — **no branching in the engine**. Sketch:

```ruby
# app/services/ai/generate.rb  — ILLUSTRATIVE
module Ai
  class Generate
    def self.call(type:, parent:, transcript:, user:)
      agent  = AiObject::Registry.agent_for(type, :generate)
      attrs  = agent.with(anchor: parent, user:).ask(transcript).content   # schema => parsed Hash
      record = AiObject::Registry.build(type, parent, attrs)               # parent.articles.build(...)
      record.save! && record
    end
  end
end
```

`Refine` is the same minus the build step (it updates the passed object); both call the shared
extractor + context composer.

## 7. Generalization 4 — the chat stays the universal substrate

`Chat` never references a domain type:

```ruby
class Chat < ApplicationRecord
  acts_as_chat
  belongs_to :chattable, polymorphic: true, optional: true   # anchor: parent (generate) | self (refine)
  enum :purpose, AiObject::Registry.purposes, validate: { allow_nil: true }   # auto-populated
end
```

The anchor is the *parent* for generate, the *object itself* for refine. This was already generic in
the content-app version — it's the **registry and context** that needed abstracting, not the chat.

## 8. What an app author actually writes to add an AI object

The full surface to make a new resource AI-generatable **and** refinable:

1. **Model** — `include AiObject; ai_object schema:, context:, parent:` (one block).
2. **Schema** — `app/schemas/<object>_schema.rb` (the structural directive).
3. **Prompts (optional)** — `app/prompts/<object>/{generate,refine}_agent/instructions.txt.erb` for
   custom voice; omit to use the generic agent.
4. **Tools (optional)** — only when a fact is large/optional (RAG, lookups), user-scoped.

No engine edits, no registry rows, no controller branching. Generate and refine come for free.

---

## 9. Cross-cutting (engine responsibilities)

- **Authorization** — anchors resolve through the signed-in user's associations (foreign id → 404);
  tools take `user:` in their constructor so they can only read that user's data.
- **Two LLM moments** — the *conversation* is a persisted, streamed `Chat` (Turbo + `ChatStreamJob`,
  tool calls auto-persisted); the *commit* is a one-shot `schema` `ask` on a transient chat so it
  never pollutes the transcript. Keep a prompt-JSON fence-strip fallback for endpoints that ignore
  `response_format`.
- **Per-tenant** — `RubyLLM.context { … }` for bring-your-own-key / per-user model choice.
- **Cost/tokens** — persisted and normalized per message for quotas and dashboards.

---

## 10. Testing

Because facts and directives are separated and ordering is declared, most of the engine is
deterministic without the network:

- **Context composer** — provider set + segment order per purpose, no LLM call.
- **Registry/dispatch** — each `purpose` resolves to the right verb/type/agent; unknown → handled.
- **Schema round-trip** — `schema` returns a parsed Hash; fallback covers ignored `response_format`.
- **Tools** — invoked when relevant, ignored otherwise, strictly user-scoped.
- **Live-LLM smoke** — one happy-path generate + refine per object; keep these few.

---

## 11. Summary

The engine is: **polymorphic chat + `purpose` dispatch + a two-verb (generate/refine) extraction loop
+ pluggable context providers.** Apps extend it by **declaring nouns and schemas** — never by editing
the engine. The generate/refine symmetry is not special-cased; it falls out of the rule that
*self-context exists only once the object does.*
